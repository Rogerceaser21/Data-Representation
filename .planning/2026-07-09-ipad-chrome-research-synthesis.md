## The one-paragraph answer (why Chrome, not Safari)

Chrome on iPadOS is not its own browser engine: Apple forces it to embed the system WKWebView, and Chrome's toolbar is native app UI wrapped AROUND that web view. Safari animates its toolbar as a fine-grained, per-frame viewport inset with fixed/sticky elements special-cased to follow; WKWebView gives Chrome no such hooks, so on every toolbar or keyboard transition Chrome coarsely resizes the whole web view, late and coalesced (Chromium engineer statement, github.com/bokand/URLBarSizing). Everything the R3 form anchors to (`position:fixed`, `window.innerHeight`, `100vh`) is therefore unstable in Chrome exactly during transitions, and Chrome's toolbar can repaint OVER page content at any time (page z-index is meaningless against app-layer chrome). On top of that sit live WebKit bugs Chrome triggers more often than Safari: additive keyboard contentInset with buggy cleanup, `overflow:hidden` defeat when the visual viewport is smaller than the layout viewport (bug 240860), and the iOS 26 stuck `visualViewport.offsetTop` regression (bug 297779).

## Per symptom

### S1 · Infinite scroll into the void

- **Root cause:** two stacked mechanisms. Baseline: `body { min-height:100vh }` sizes to the LARGEST viewport, guaranteeing a small permanent scrollable overhang whenever Chrome's toolbar is expanded (bram.us/100vh, gatsby#14590). Main event: WKWebView adds the keyboard height to the scroll view's bottom contentInset ADDITIVELY on input focus (jamesreggio gist), its dismissal-time cleanup is a historically buggy path that leaves the extra scrollable extent behind (WebKit 192564, fek.io: does not self-heal without reload), and our pad open/close `scrollTo` calls execute against that corrupted bookkeeping (WebKit 182710). `overscroll-behavior` cannot clamp the root scroller in Chrome iOS (ebidel/demos#6). 6-step reproduction theory in the findings doc, Task 4.
- **Fix (v0.57):** (i) body `min-height: 100svh` (fallback `100vh` for pre-15.4), removing the baseline overhang; (ii) never issue programmatic scrolls while keyboard state is in flux: `openPad`/`openPadView` first blur any focused element, wait a settle beat (~2 rAF + 100ms), THEN save scrollY and proceed; (iii) after every `focusout` from a text field, a corrective micro-nudge `scrollBy(0,-1); scrollBy(0,1)` after settle to force WKWebView to re-settle its inset bookkeeping (documented workaround, iifx.dev); (iv) the in-document swap (see S3) removes the scrollable document entirely while the pad is open, which is when the worst corruption was being triggered.
- **Risk:** micro-nudge is a heuristic against a system-component bug; if iOS fixes 297779/inset cleanup it becomes harmless. `svh` makes the page slightly shorter when the toolbar collapses (background fills, no layout break).
- **On-device test (Chrome + Safari):** type in Weakness textarea, dismiss keyboard via accessory strip, scroll to bottom: page must stop at the footer (no void). Repeat after a pad open/close cycle.

### S2 · Bottom controls mid-screen

- **Root cause:** the v0.56 `--vv-shift` formula is the community-standard recipe, but (i) its reference `window.innerHeight` is unreliable in WKWebView (WebKit 170595, open since 2017, confirmed in Chrome iOS 2025); (ii) it reads `visualViewport` synchronously in the event tick, a documented stale-read race (WebKit 237851, fix = deferred read); (iii) while the keyboard is up, iOS pans the visual viewport and fixed elements effectively behave like `position:static` regardless of any transform (CSSWG 7475); (iv) the iOS 26 regression leaves `offsetTop` stuck ~24px after dismissal = permanent phantom shift. The `gap > 200px` guard cannot save a state where the whole anchor system is drifting.
- **Fix (v0.57):** two layers. Layer 1, hide-while-typing: on `focusin` of any text field add `body.kb-open` which fades out the three bottom controls; reshow on `focusout` after ~350ms settle. In every keyboard state the controls are simply absent, so nothing can land mid-screen (and observers do not need Reset/Save while typing). Layer 2, harden the pin for non-keyboard states: replace the `window.innerHeight` reference with `document.documentElement.clientHeight` captured per-update, defer all reads by double-rAF, clamp gap to [0, 120], keep the 3-place transform contract. The unmeasured accessory-strip-only state is covered by the instrumentation task below.
- **Risk:** brief control fade in/out around typing (cosmetic); accessory strip WITHOUT focus (hardware keyboard idle state) still unmeasured, diagnostics will tell.
- **On-device test:** Chrome, focus a comment field mid-form, type, dismiss: controls hidden while typing, reappear glued to the bottom; scroll the full form up and down: controls never detach from the bottom edge.

### S3 + N1 · Pad: scroll indicator, form underneath, toolbar covering pad buttons

- **Root cause:** the pad is a fixed overlay over a still-laid-out, still-scrollable form document. `containScroll` only swallows gestures targeted at the pad; the DOCUMENT still scrolls via other routes, its indicator paints along the pad's right edge, scroll deltas invite Chrome's toolbar to re-expand, and the re-expanded toolbar paints OVER the pad's top buttons because it is app-layer UI. `inset:0` composites short against the shifted layout viewport, exposing the form beneath the pad's bottom edge (iOS 26 bug 297779 makes full-screen overlays not cover the full viewport).
- **Fix (v0.57), the architecture decision:** **in-document swap** (Igor's "another page" instinct, implemented without navigation). While the pad (or the pad-view lightbox) is open: hide `#r3-form`, `.action-bar`, theme toggle and admin cog (`display:none` via a `body.pad-open` class) AND collapse `body` sizing (`body.pad-open { min-height:0; padding:0 }`), so the pad is the only laid-out content and the document has zero scrollable extent. No extent = no scroll indicator, nothing to scroll under the pad, no scroll deltas to wake Chrome's toolbar. Close = restore display, then `scrollTo(0, savedY)` against an identical layout (strongest possible restore guarantee), then run the existing v0.55 background extraction. Verified against the code: TomSelect, autosave, Konva sizing, `extractAndApplySilently` (writes to hidden fields work), photo inputs all survive; the pad itself keeps `position:fixed` and all internals unchanged.
- **Why not a true separate page:** the extraction pipeline writes directly into the same-document form (would need a cross-page re-plumb of ~1200 lines), a third StatiCrypt artifact with unverified gate carryover, and, decisively, page navigation can surface Chrome's NATIVE error page on a WiFi hiccup, violating hard rule 12. The swap gives the same isolation with zero navigation.
- **Risk:** three on-device unknowns: non-scroll toolbar triggers (edge swipe) may still re-expand Chrome's toolbar over the pad top (mitigation if seen: extra top padding on `.pad-top` in Chrome via UA check is NOT preferred; accept or revisit); stale keyboard inset from typing immediately before pad open (mitigated by the blur + settle in S1 fix); TomSelect visibility-toggle quirks (check dropdown reopen after pad close).
- **On-device test:** scroll mid-form, type in a field, then immediately open the pad from the Weakness pill: pad fills the screen, NO scroll indicator, NO form visible at any edge, toolbar stays put while drawing (including Pencil strokes + page flips); Done returns to the exact scroll spot; teacher-viewer lightbox same drill.

## Locked-constraint cross-check

- No Fullscreen API: none used. No body repositioning: only `display:none` on siblings + `min-height`/`padding` collapse; body is never `position:fixed`, never offset. Exact scroll restore: strengthened (nothing can move scroll while the pad is open). Hard rule 12: all fixes are CSS/JS state toggles, no navigation, no new network dependency, no error UI. StatiCrypt single-file build: unchanged. Konva/Pencil ink: pad internals untouched. `--vv-shift` 3-place transform contract: kept (values computed more robustly; controls additionally hidden during keyboard states). v0.55 auto-extraction contract: unchanged (runs on close, writes work on visible fields after restore).
