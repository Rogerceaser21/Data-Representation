### 3. Backfill: forward each inspector their missed records (IN PROGRESS)
Forwarding from the admin.user mailbox copies. Mechanics that WORK and must be
matched: build a MIME (`email.mime.text.MIMEText` html), **To = the inspector
only, NO Cc, NO Bcc**, Subject `Fwd: <original subject>`, body = short note +
"---------- Forwarded message ----------" header block + the **original HTML
part** (base64url-decoded from the original message's `text/html` part), then
`gws gmail users messages send --json '{"raw": <base64url>}'`. Note text used:
"Hi <First>, Forwarding your copy of the R3 observation record below for your
records. Kind regards, Igor".

**Correct "missed" signal = the original record email's `Cc` header is EMPTY.**
Do NOT match against the Inspectors-tab email string (it can have typos, e.g.
Dave). Records already delivered have a non-empty Cc (the system CC'd the right
address when the lookup worked) or an existing `Fwd:` to that inspector.

## Hard-won rules (do not violate)

- **Confidential observation data goes ONLY to the inspector who did it**, never
  to the inspected teacher. Every forward: single `To`, no Cc/Bcc. Verify
  recipients after sending (we did, via the To/Cc/Bcc headers).
- **Present a plan and await approval before outward/irreversible actions**; Igor
  uses "PLEASE TELL ME WHAT YOU THINK… AWAIT MY APPROVAL". He also wants a final
  confirm even when it seems obvious. For batch sends he was happy after a 1-item
  test (Lana) then "go".
- **No error UI ever** in the R3 form (hard rule 12). **No em/en dashes** anywhere.
- **Back up a Sheet tab (duplicateSheet) before destructive edits.**
- Re-derive facts from data, don't trust a stale tab value (the Dave typo bit us;
  the cc-empty signal is the truth).
